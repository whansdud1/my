## 1. 개요 (Overview)

대학생들이 팀 프로젝트 진행 시 겪는 **팀 구성의 어려움**과 **갈등 문제**를 해결하기 위한 웹 기반 팀 매칭 플랫폼이다. 학생의 전공, 일정, 협업 성향, 선호 역할, 과거 협업 실적 데이터를 분석하여 최적의 팀을 자동 추천하고, 프로젝트 진행 중 협업 데이터를 수집·분석해 위험 신호를 표시하며, 종료 후 동료 평가와 AI 분석을 결합해 사용자 평점을 갱신함으로써 향후 팀 매칭 정확도를 지속적으로 개선한다.

본 명세서는 비즈니스 이해관계자(학생, 교수, 학교 행정담당자, 운영자)가 검토할 수 있도록 **사용자 가치와 시스템이 해야 할 일**에 초점을 맞추고 있으며, 구현 기술 스택은 다루지 않는다.

## 2. User Scenarios & Testing *(mandatory)*

### 2.1 Primary User Story

**김민서(경영학과 3학년) 시점**

민서는 학과 공모전 참가를 위해 팀원을 구성해야 한다. 기존에는 친한 친구 위주로 팀을 짰지만, 매번 역할이 한쪽에 치우치거나 일정이 맞지 않아 갈등이 잦았다.

1. 민서는 플랫폼에 가입하면서 성별·학년·전공·선호 역할(PPT·자료조사)·가능 협업 시간대(주로 야간)·협업 성향 설문에 응답한다.
2. 민서가 "경영학과 공모전" 프로젝트를 생성하고 모집 인원·필요 역량·일정·작업 시간 선호도를 입력한다.
3. 시스템이 민서를 포함한 협업 가능 시간이 겹치고 부족한 역할(발표·기획·디자인·일정관리)을 채울 수 있는 후보를 점수와 함께 추천한다.
4. 민서는 후보 프로필과 별점·리뷰 요약을 확인한 뒤 팀원을 확정한다.
5. 팀은 Zoom·Discord·Notion·Google Workspace 중 사용할 협업 도구를 연결한다. 시스템은 회의 참여, 작업 업로드, 마감 준수, 메시지 응답 속도를 수집한다.
6. 한 팀원의 자료 공유가 지연되자 대시보드에 **협업 위험 신호**가 표시되고, 팀장은 사전에 조치를 취한다.
7. 프로젝트 종료 후 팀원 간 동료 평가가 진행되고, AI는 활동 데이터를 수치화해 종합 평점을 산출, 각 학생 프로필에 반영한다.
8. 다음 학기 다른 프로젝트에서 민서를 추천 후보로 띄울 때 이 평점과 리뷰가 가중치로 작용한다.

### 2.2 Acceptance Scenarios

| # | Given (전제) | When (행위) | Then (기대 결과) |
|---|---|---|---|
| AS-01 | 신규 학생이 회원가입 화면에 진입함 | 성별·학년·전공·선호 역할·협업 성향 설문을 작성하고 제출함 | 프로필이 저장되고 매칭 후보 풀에 포함된다 |
| AS-02 | 학생이 프로젝트(공모전·수업·자율)를 등록함 | 모집 인원·필요 역할·일정·선호 시간대를 입력함 | 시스템이 적합도 점수 순으로 후보 목록을 제시한다 |
| AS-03 | 추천 후보 리스트가 노출됨 | 학생이 한 명의 프로필을 클릭함 | 평점·리뷰 요약·강점 역할·가능 시간이 시각화되어 표시된다 |
| AS-04 | 팀이 확정됨 | 협업 플랫폼(Zoom·Notion 등)을 연결함 | 회의 참여·업로드·마감·응답 데이터가 자동 수집·기록된다 |
| AS-05 | 한 팀원의 응답 속도와 업로드 빈도가 임계치 이하로 떨어짐 | 협업 데이터 분석 모듈이 임계값을 감지함 | 팀 대시보드에 위험 신호와 권장 조치가 표시된다 |
| AS-06 | 프로젝트가 종료됨 | 팀원이 상호 평가(기여도·소통·책임감·만족도) 항목을 채움 | 평가가 익명 처리되어 저장되고 평점에 반영된다 |
| AS-07 | 동료 평가가 모두 제출됨 | AI가 활동 데이터와 동료 평가를 결합해 종합 평점을 산출함 | 학생별 프로필 평점·별점·리뷰가 갱신된다 |
| AS-08 | 동일 평가자가 특정 학생에게만 반복적으로 낮은 점수를 부여함 | 시스템이 비정상 평가 패턴을 감지함 | 관리자에게 검토 요청 알림이 전송되고 해당 평가는 보류 상태로 표시된다 |
| AS-09 | 팀 정원이 초과된 인기 프로젝트가 있음 | 프리미엄 구독자가 지원함 | 프리미엄 구독자가 우선 매칭 후보로 표시된다 |
| AS-10 | 학생이 자신의 프로필 데이터 삭제를 요청함 | 데이터 삭제 요청을 제출함 | 시스템이 식별 데이터를 삭제하고 평가 이력은 익명 통계로만 보존한다 |

### 2.3 Edge Cases

- **EC-01 매칭 후보 부족**: 특정 시간대·전공 조합으로 추천 후보가 N명 미만일 때 시스템은 차선 후보(시간 겹침 비율을 점진적으로 완화)를 제시한다.
- **EC-02 동시 매칭 충돌**: 두 프로젝트가 동시에 같은 학생을 확정하려 할 때 먼저 확정 요청한 쪽이 우선되며 다른 쪽엔 차선 후보가 제안된다.
- **EC-03 협업 도구 미연동 팀**: 외부 협업 도구 연결을 거부하는 팀은 수동 활동 기록(체크인·작업 인증) 모드로 전환한다.
- **EC-04 일정·역할 충돌**: 학생이 동시에 2개 이상의 진행 중 프로젝트에 속하고 부담 한도(시스템 기본값 동시 3건)를 초과하면 신규 매칭 추천에서 일시 제외된다.
- **EC-05 평가 미제출**: 프로젝트 종료 후 평가 마감(기본 14일) 내 미제출 시 자동 리마인더 발송, 마감 후엔 활동 데이터만으로 잠정 평점이 산출된다.
- **EC-06 악의적 평가 보복 의심**: 직전 프로젝트에서 갈등을 겪은 사용자의 평가가 통상 분포에서 크게 벗어나면 가중치 감소 또는 관리자 검토로 자동 전환된다.
- **EC-07 신규 사용자 평점 부재**: 평점 이력이 없는 신규 학생은 자기소개·설문 기반 신뢰 점수와 "신규" 배지를 부여하여 매칭 노출 기회를 보장한다.
- **EC-08 외부 협업 도구 API 장애**: 협업 데이터 수집이 일시 중단되면 마지막 정상 수집 시점을 표시하고 위험 신호 판정은 보류한다.
- **EC-09 GDPR/개인정보 요청**: 학생이 자기 데이터 열람·정정·삭제를 요청할 시 30일 이내 처리 가능해야 한다.
- **EC-10 알고리즘 편향 누적**: 동일 학생군에게만 매칭이 집중되면 다양성 가중치를 자동 상향 적용해 노출이 적은 학생을 후보군에 포함시킨다.

## 3. Requirements *(mandatory)*

### 3.1 Functional Requirements

#### FR-A. 회원·프로필 관리
- **FR-A1**: 시스템은 학생이 이메일(대학 도메인 우선)과 비밀번호로 회원가입할 수 있어야 한다. 본인 검증은 인증 메일 링크 클릭으로 완료된다.
- **FR-A2**: 시스템은 가입 시 다음 항목을 필수로 수집해야 한다: 성별, 학년, 소속 학과/전공, 선호 역할(다중 선택), 협업 성향 설문(MBTI 유형 또는 협업 스타일 설문 12문항).
- **FR-A3**: 시스템은 선택 항목으로 수강 과목, 주간 가능 시간표, 협업 가능 시간대, 자기소개를 수집해야 한다.
- **FR-A4**: 시스템은 학생이 본인 프로필을 언제든 조회·수정·삭제할 수 있도록 해야 한다.
- **FR-A5**: 시스템은 사용자 역할을 "학생", "교수", "관리자"로 구분하여 권한을 다르게 부여해야 한다.

#### FR-B. 프로젝트 등록·매칭
- **FR-B1**: 학생 또는 교수는 프로젝트(공모전·수업 과제·자율 모임)를 등록할 수 있어야 하며, 모집 인원·필요 역할·일정·작업 시간 선호·설명을 입력한다.
- **FR-B2**: 시스템은 등록된 프로젝트 요건과 사용자 프로필을 비교하여 적합도 점수를 산출하고 상위 후보 목록을 추천해야 한다.
- **FR-B3**: 적합도 점수에는 다음 요소가 반영되어야 한다: 역할 균형, 가능 시간 겹침 비율, 협업 성향 호환도, 평점·리뷰 가중치, 다양성 가중치.
- **FR-B4**: 추천 결과에는 각 후보의 강점 역할, 가능 시간, 평점, 별점, 리뷰 요약이 함께 제시되어야 한다.
- **FR-B5**: 프로젝트 등록자는 후보 중에서 팀원을 직접 선택·초대할 수 있어야 하며, 초대받은 학생은 수락/거절할 수 있어야 한다.
- **FR-B6**: 시스템은 학생이 동시에 진행 가능한 프로젝트 수 한도(기본 3건)를 적용하고 초과 시 추천에서 제외해야 한다.

#### FR-C. 협업 데이터 수집·분석
- **FR-C1**: 시스템은 외부 협업 도구(Zoom·Discord·Notion·Google Workspace)와 사용자 동의 하에 연동되어 협업 활동 데이터를 수집해야 한다.
- **FR-C2**: 수집 항목은 최소 다음을 포함해야 한다: 회의 참여 횟수, 작업/문서 업로드 빈도, 마감 준수 여부, 메시지 응답 회신 속도.
- **FR-C3**: 시스템은 팀별 협업 대시보드에서 위 지표를 시각화해야 한다.
- **FR-C4**: 시스템은 지정 임계값(예: 응답 지연, 업로드 정체) 초과 시 **협업 위험 신호**를 팀 대시보드와 팀장에게 알림으로 제공해야 한다.
- **FR-C5**: 학생은 협업 데이터 수집 범위를 본인 설정에서 조정·중단할 수 있어야 한다.

#### FR-D. 일정 조율
- **FR-D1**: 시스템은 팀원의 가용 시간표를 자동 비교해 공통 가능 시간 후보를 제안해야 한다.
- **FR-D2**: 시스템은 외부 캘린더(Google Calendar 등)와 동기화 가능해야 한다.
- **FR-D3**: 시스템은 마감일·작업 일정·회의 일정을 팀원 모두에게 공유하고 리마인더를 발송해야 한다.

#### FR-E. 평가·리뷰
- **FR-E1**: 프로젝트 종료 시 시스템은 팀원에게 동료 평가 요청을 발송해야 한다. 평가 항목은 기여도·의사소통·책임감·협업 만족도 + 자유 코멘트로 구성된다.
- **FR-E2**: 동료 평가는 익명 처리되어야 하며, 평가자/피평가자는 식별되지 않아야 한다.
- **FR-E3**: AI 분석 평점은 활동 데이터(FR-C2)를 기반으로 역할별 진행도·완성도를 수치화해 산출해야 한다.
- **FR-E4**: 종합 평점 = (동료 평가) × W1 + (AI 분석 평점) × W2 형태의 가중 평균으로 산출되며 사용자 프로필에 별점·리뷰 요약 형식으로 노출되어야 한다.
- **FR-E5**: 평가 마감(기본 14일) 내 미제출 시 시스템은 자동 리마인더를 발송하고 마감 후엔 활동 데이터 기반 잠정 평점을 적용해야 한다.

#### FR-F. 추천 시스템 학습·개선
- **FR-F1**: 시스템은 종료된 프로젝트 결과(완수/중도 이탈)와 만족도를 학습 데이터로 활용해 추천 알고리즘을 주기적으로 갱신해야 한다.
- **FR-F2**: 반복적으로 우수한 성과를 낸 패턴은 가중치를 상향하고 부정 성과 패턴은 하향 조정해야 한다.
- **FR-F3**: 시스템은 알고리즘 편향을 방지하기 위해 일정 기간 매칭 노출이 적은 학생에 대해 다양성 가중치를 자동 상향해야 한다.

#### FR-G. 허위·악성 평가 탐지
- **FR-G1**: 시스템은 평가 분포 이상치(특정 평가자→특정 피평가자 반복 저평가, 활동 데이터와 평가 점수 간 큰 괴리)를 자동 감지해야 한다.
- **FR-G2**: 이상 감지 시 시스템은 해당 평가를 "검토 보류" 상태로 표시하고 관리자에게 알림을 보내야 한다.
- **FR-G3**: 관리자는 검토 결과에 따라 평가를 승인/철회하고 악의 평가자에게 경고·제재(추천 가중치 감소·계정 정지)를 적용할 수 있어야 한다.

#### FR-H. 알림·커뮤니케이션
- **FR-H1**: 시스템은 매칭 초대, 팀 확정, 일정 변경, 위험 신호, 평가 요청, 관리자 조치를 인앱·이메일로 알림해야 한다.
- **FR-H2**: 시스템은 팀 내 기본 채팅 기능을 제공해야 한다(무료 기능).

#### FR-I. 수익화·구독 관리
- **FR-I1**: 시스템은 무료/프리미엄 구독 계층을 운영해야 한다. 무료 계층은 팀 매칭·공통 일정 분석·기본 일정 관리·팀 채팅을 포함한다.
- **FR-I2**: 프리미엄 계층은 AI 기반 일정 최적화, 심화 협업 분석, 인기 프로젝트 우선 매칭, 광고 제거를 제공해야 한다.
- **FR-I3**: 시스템은 학교/기관 단위 제휴 계약 관리 기능을 운영자에게 제공해야 한다.

#### FR-J. 개인정보·데이터 거버넌스
- **FR-J1**: 시스템은 수집·보관·이용 목적을 명시한 동의 절차를 회원가입 시 제공해야 한다.
- **FR-J2**: 시스템은 학생이 데이터 열람·정정·삭제를 요청할 수 있는 창구를 제공하고 30일 이내 처리해야 한다.
- **FR-J3**: 시스템은 민감 데이터(평가·협업 활동)에 대해 암호화 저장·전송을 보장해야 한다.
- **FR-J4**: 평가 이력은 식별자가 분리된 익명 통계로 변환 후 보존하며, 원본 식별 데이터는 사용자 요청 시 삭제 가능해야 한다.

#### FR-K. 운영·관리자 기능
- **FR-K1**: 관리자는 신고된 평가, 이상 감지 케이스, 사용자 제재 이력을 통합 콘솔에서 처리할 수 있어야 한다.
- **FR-K2**: 관리자는 매칭 알고리즘 가중치·임계값을 조정할 수 있어야 한다(추후 정책 변경 대응).
- **FR-K3**: 관리자는 사용자/프로젝트 통계 대시보드를 통해 서비스 운영 지표(매칭 성공률, 활성 사용자 수, 평가 완료율 등)를 확인할 수 있어야 한다.

### 3.2 Clarifications (Resolved)

> 아래 항목은 **/speckit.clarify** (2026-05-22) 에서 이해관계자 결정으로 확정되었다.

- **CQ-01 → 혼합형 (학생 자율 + 수업 연계)** — 1차 출시부터 두 진입점을 모두 지원한다.
  - 학생 자율: 학생이 직접 `Project` 를 생성하고 팀원을 모집한다(유형 = `자율`/`공모전`).
  - 수업 연계: 교수 권한 계정이 수업/과제 단위로 `Project(유형 = 수업)` 를 등록하면 학생을 자동/반자동으로 매칭한다.
  - **권한 모델**: `Role ∈ {Student, Professor, Admin}`. 교수 권한은 학생 신분 검증과 별개 경로(기관 제휴/관리자 수동 승인)로 부여.
  - **온보딩 흐름**: 가입 시 역할 선택 분기(`Student` / `Professor`). 교수는 별도 승인 큐를 거친다.
- **CQ-02 → 이메일 도메인(.ac.kr) + 학생증 업로드 2단계** — 가입 시 1단계로 대학 이메일 도메인 인증(이메일 OTP/링크), 활성화 후 2단계로 학생증 사본 업로드 + 관리자 수동 승인(또는 OCR 자동 보조). 미승인 상태는 매칭 후보 풀 진입이 제한된다.
  - 보조 도메인 화이트리스트는 `partnerships` 테이블의 제휴 기관에서 관리.
  - 학생증 이미지는 암호화 저장(서버 측), 승인 후 N일 내 자동 폐기.
- **CQ-03 → 최대 집합 (메시지 본문 분석 포함)** — 1차 출시부터 메타데이터(회의 참석/시간, 업로드 빈도)와 메시지 본문(Zoom/Discord/Slack/Notion/Google Workspace) 까지 수집·분석한다.
  - **전제 조건**:
    1. OAuth 동의 화면에 **항목별** 수집 범위·보존 기간·목적을 명시적으로 표시.
    2. 본문은 **AI 분석 입력 후 즉시 폐기**(원문 비저장) 또는 사용자가 선택 시 N일 단기 보관.
    3. DSR(열람/정정/삭제) 30일 SLA 가 본문 분석 결과에도 적용.
    4. 개인정보 영향평가(PIA) 결과 위험도가 임계치 초과인 경우 자동으로 **중간 집합**(메타+요약)으로 다운그레이드되는 정책 스위치 보유.

## 4. Success Criteria *(mandatory)*

### 4.1 Measurable Outcomes

**사용자 가치 지표**

- **SC-01**: 신규 학생이 회원가입을 시작해서 첫 매칭 추천을 받기까지 **평균 10분 이내**에 완료할 수 있어야 한다.
- **SC-02**: 학생의 80% 이상이 추천된 상위 5명 중 최소 3명을 "협업 가능"으로 평가해야 한다.
- **SC-03**: 매칭으로 결성된 팀의 프로젝트 완수율이 기존 자율 구성 대비 **최소 20% 이상 향상**되어야 한다.
- **SC-04**: 팀원 협업 만족도(5점 척도) 평균이 **4.0 이상**이어야 한다.
- **SC-05**: 위험 신호 알림을 받은 팀의 90% 이상이 알림 후 7일 이내에 위험 지표를 개선해야 한다.

**시스템 신뢰성 지표**

- **SC-06**: 시스템은 동시 사용자 **10,000명** 규모에서 매칭 추천을 5초 이내에 응답해야 한다.
- **SC-07**: 핵심 기능(가입·매칭·일정·평가)에 대한 가용성은 월 단위 **99.5% 이상**을 유지해야 한다.
- **SC-08**: 외부 협업 도구 연동 데이터 수집 성공률 **95% 이상**(최근 24시간 기준).

**공정성·안전성 지표**

- **SC-09**: 동일 학생군 매칭 집중도(상위 10% 학생에게 노출된 매칭 비율)가 **40% 이하**로 유지되어야 한다.
- **SC-10**: 허위·악성 평가 자동 감지 정확도(precision) **85% 이상**, 재현율(recall) **70% 이상**.
- **SC-11**: 개인정보 열람·정정·삭제 요청은 **30일 이내 100%** 처리되어야 한다.

**비즈니스 지표**

- **SC-12**: 출시 후 6개월 이내 누적 가입 학생 **5,000명**, 누적 결성 팀 **1,000건** 이상.
- **SC-13**: 프리미엄 구독 전환율 **활성 사용자의 5% 이상**.
- **SC-14**: 대학·기관 제휴 계약 **5건 이상 체결**.

## 5. Assumptions

본 명세서는 다음 가정을 기반으로 작성되었다. 각 가정은 추후 변경 가능하다.

- **A-01** 1차 출시 대상은 국내 4년제 대학생이며, 모바일 우선 반응형 웹 형태로 제공한다.
- **A-02** 사용자 인증은 이메일·비밀번호 기반이며 **대학 이메일 도메인(.ac.kr) 인증 + 학생증 업로드 2단계 승인**을 필수로 한다 (CQ-02 확정, 2026-05-22). 대학 SSO 연계는 향후 제휴 단계에서 확장.
- **A-03** 협업 데이터는 사용자가 직접 OAuth로 연결한 도구에서만 수집되며, 미연동 시 수동 기록 모드가 기본이다.
- **A-04** 평가 가중치 W1(동료 평가):W2(AI 분석)의 초기값은 60:40으로 설정하며 운영 데이터에 따라 조정한다.
- **A-05** 학생이 동시에 진행 가능한 프로젝트 한도 기본값은 3건, 평가 마감 기본 14일이다.
- **A-06** 결제 처리는 외부 결제대행사(PG)를 통해 위임하며 카드 정보를 시스템 내에 저장하지 않는다.
- **A-07** 학교 LMS 연계는 1차 출시 범위 외이며, 별도 제휴 계약 시 단계적 도입한다.
- **A-08** 광고 노출은 무료 사용자 한정이며 학습/취업/스터디 관련 카테고리로 제한한다.
- **A-09** 위험 신호 임계값은 통계 기반 동적 임계치를 기본으로 하며 관리자 조정이 가능하다.
- **A-10** 모든 평가는 익명 처리되지만 이상 패턴 조사 시 관리자는 식별 가능한 권한을 가진다(감사 로그 남김).
