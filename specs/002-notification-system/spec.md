## 1. 개요 (Overview)

UniTeam 팀 매칭 플랫폼에서 발생하는 주요 이벤트(매칭 추천, 팀 확정, 협업 위험 신호, 평가 요청, 일정 변경, 관리자 검토 등)를 사용자에게 **적시에, 적절한 채널로** 전달하는 통합 알림 시스템이다. 사용자는 알림 종류별로 수신 여부와 채널(인앱·이메일·푸시)을 직접 설정할 수 있으며, 시스템은 중복·과다 알림을 방지하고 읽음 상태를 관리한다.

본 명세서는 비즈니스 이해관계자(학생, 팀장, 운영자)가 검토할 수 있도록 **사용자 가치와 시스템이 해야 할 일**에 초점을 맞추며, 구현 기술 스택은 다루지 않는다.

---

## 2. User Scenarios & Testing *(mandatory)*

### 2.1 Primary User Story

**김민서(경영학과 3학년) 시점**

민서는 여러 팀 프로젝트에 동시에 참여 중이라, 중요한 일정이나 팀 활동을 놓치는 일이 잦았다.

1. 민서가 프로젝트를 생성하자, 적합한 후보가 추천되는 즉시 **인앱 알림**과 **이메일**로 "새 매칭 후보 3명이 도착했습니다" 알림을 받는다.
2. 한 후보에게 팀 합류를 요청하자, 상대방에게 알림이 전달되고 상대가 수락하면 민서에게 "○○○님이 팀에 합류했습니다" 알림이 온다.
3. 팀 활동 중 한 팀원의 응답이 지연되어 협업 위험 신호가 발생하면, 팀장인 민서에게 **우선순위 높은 알림**이 표시된다.
4. 프로젝트 종료 후 동료 평가 마감 3일 전, 민서는 "평가가 아직 제출되지 않았습니다" 리마인더를 받는다.
5. 민서는 알림이 너무 많다고 느껴 설정에서 "일정 변경" 알림은 인앱만, "매칭" 알림은 이메일까지 받도록 채널을 조정한다.
6. 민서는 알림 센터에서 지난 알림 목록을 확인하고, 읽지 않은 알림만 필터링하거나 전체를 읽음 처리한다.

### 2.2 Acceptance Scenarios

| # | Given (전제) | When (행위) | Then (기대 결과) |
|---|---|---|---|
| AS-01 | 학생이 프로젝트를 생성하고 매칭이 완료됨 | 시스템이 추천 후보를 산출함 | 프로젝트 생성자에게 매칭 완료 알림이 인앱·이메일로 전송된다 |
| AS-02 | 학생이 다른 학생에게 팀 합류를 요청함 | 요청이 제출됨 | 대상 학생에게 합류 요청 알림이 전송되고 수락/거절 동작이 포함된다 |
| AS-03 | 합류 요청 대상자가 수락/거절함 | 응답이 처리됨 | 요청자에게 결과 알림이 전송된다 |
| AS-04 | 팀 협업 데이터가 위험 임계치를 초과함 | 분석 모듈이 위험 신호를 감지함 | 팀장에게 우선순위 높음(긴급) 알림이 전송된다 |
| AS-05 | 프로젝트가 종료되고 평가 마감이 다가옴 | 마감 N일 전 시점이 도래함 | 미제출 평가자에게 리마인더 알림이 전송된다 |
| AS-06 | 사용자가 알림 센터에 진입함 | 알림 목록을 조회함 | 최신순 정렬·읽음/안읽음 구분된 목록과 안읽음 개수가 표시된다 |
| AS-07 | 사용자가 알림을 클릭함 | 알림을 선택함 | 해당 알림이 읽음 처리되고 관련 화면(프로젝트·평가 등)으로 이동한다 |
| AS-08 | 사용자가 알림 설정 화면에 있음 | 알림 종류별 채널 on/off를 변경하고 저장함 | 이후 해당 종류 알림은 설정된 채널로만 전송된다 |
| AS-09 | 동일 이벤트가 짧은 시간에 반복 발생함 | 시스템이 중복 알림 조건을 감지함 | 중복 알림이 묶이거나 억제되어 1건으로 전달된다 |
| AS-10 | 사용자가 전체 알림 수신을 비활성화함(방해 금지) | 방해 금지 모드를 켬 | 긴급 알림을 제외한 알림이 보류되고 모드 해제 시 요약 제공된다 |

### 2.3 Edge Cases

- **EC-01 채널 전송 실패**: 이메일·푸시 전송이 실패하면 인앱 알림은 보존되고, 실패 채널은 정해진 횟수까지 재시도 후 중단한다.
- **EC-02 수신 거부 사용자**: 사용자가 모든 채널을 끈 종류의 알림은 전송하지 않되, 법적·계정 보안 관련 필수 알림은 예외로 항상 전송한다.
- **EC-03 대량 동시 알림**: 단일 이벤트로 다수 사용자에게 알림이 발생할 때(예: 공지) 전송이 지연되더라도 누락 없이 순차 처리된다.
- **EC-04 읽음 상태 동기화**: 여러 기기에서 접속한 사용자가 한 기기에서 읽음 처리하면 다른 기기에도 반영된다.
- **EC-05 오래된 알림 보관**: 일정 기간(기본 90일) 경과한 알림은 보관 처리되어 기본 목록에서 숨겨진다.
- **EC-06 삭제된 대상 참조**: 알림이 가리키는 프로젝트·평가가 삭제된 경우 클릭 시 "더 이상 존재하지 않는 항목" 안내를 표시한다.
- **EC-07 잘못된 연락처**: 이메일 주소가 유효하지 않으면 해당 채널 전송을 중단하고 사용자에게 인앱으로 연락처 갱신을 요청한다.
- **EC-08 시간대 차이**: 야간 시간대(기본 22:00~08:00)에는 긴급 외 푸시·이메일 발송을 다음 날 아침으로 묶어 보낸다.

## 3. Requirements *(mandatory)*

### 3.1 Functional Requirements

#### FR-A. 알림 생성·트리거
- **FR-A1**: 시스템은 정의된 이벤트(매칭 완료, 합류 요청/응답, 협업 위험 신호, 평가 요청/리마인더, 일정 변경, 관리자 검토, 시스템 공지)가 발생하면 해당 수신자에게 알림을 생성해야 한다.
- **FR-A2**: 각 알림은 종류(type), 우선순위(긴급/일반/정보), 제목, 본문, 연결 대상(deep link), 생성 시각을 포함해야 한다.
- **FR-A3**: 시스템은 알림 종류별로 기본 수신 대상(개인/팀장/팀 전체/관리자)을 정의해야 한다.

#### FR-B. 채널·전달
- **FR-B1**: 시스템은 최소 3개 채널(인앱, 이메일, 푸시)을 지원해야 한다.
- **FR-B2**: 인앱 알림은 모든 알림 종류에 대해 항상 보존되어야 한다(사용자가 알림 센터에서 확인 가능).
- **FR-B3**: 이메일·푸시 전송 실패 시 정해진 횟수만큼 재시도하고, 최종 실패는 로그로 기록해야 한다.
- **FR-B4**: 긴급(critical) 알림은 사용자의 방해 금지/야간 설정과 무관하게 즉시 전달되어야 한다.

#### FR-C. 사용자 설정
- **FR-C1**: 사용자는 알림 종류별로 채널(인앱/이메일/푸시)별 수신 여부를 켜고 끌 수 있어야 한다.
- **FR-C2**: 사용자는 방해 금지 모드와 야간 묶음 발송 시간대를 설정할 수 있어야 한다.
- **FR-C3**: 보안·계정·법적 고지 등 필수 알림은 사용자가 끌 수 없어야 한다.
- **FR-C4**: 설정 변경은 즉시 이후 알림에 반영되어야 한다.

#### FR-D. 알림 센터·읽음 관리
- **FR-D1**: 사용자는 알림 목록을 최신순으로 조회하고 읽음/안읽음으로 필터링할 수 있어야 한다.
- **FR-D2**: 시스템은 안읽음 알림 개수(배지)를 제공해야 한다.
- **FR-D3**: 사용자는 개별 알림 읽음 처리 및 전체 읽음 처리를 할 수 있어야 한다.
- **FR-D4**: 읽음 상태는 사용자의 여러 접속 환경 간에 동기화되어야 한다.

#### FR-E. 중복·과다 방지
- **FR-E1**: 시스템은 동일 사용자에게 짧은 시간 내 발생하는 동종 이벤트 알림을 묶거나 억제(rate limit)해야 한다.
- **FR-E2**: 시스템은 야간 시간대의 비긴급 알림을 묶어 다음 발송 가능 시각에 전달해야 한다.

#### FR-F. 보관·수명주기
- **FR-F1**: 일정 보관 기간이 지난 알림은 자동 보관 처리되어 기본 목록에서 숨겨야 한다.
- **FR-F2**: 시스템은 알림 전송·읽음·실패 이력을 운영 분석용으로 기록해야 한다.

### 3.3 Key Entities

| 엔티티 | 설명 | 주요 속성 |
|---|---|---|
| **Notification** | 개별 알림 인스턴스 | id, recipient, type, priority, title, body, deepLink, status(unread/read/archived), createdAt, readAt |
| **NotificationType** | 알림 종류 정의 | code, 기본 우선순위, 기본 수신 대상, 필수 여부 |
| **NotificationPreference** | 사용자별 수신 설정 | userId, typeCode, channelFlags(inApp/email/push), 방해금지 시간대 |
| **DeliveryLog** | 채널별 전송 이력 | notificationId, channel, status(sent/failed/retried), attemptCount, timestamp |

## 4. Success Criteria *(mandatory)*

### 4.1 Measurable Outcomes

- **SC-01**: 이벤트 발생 후 인앱 알림이 사용자에게 표시되기까지 95%가 5초 이내에 도달한다.
- **SC-02**: 긴급 알림의 99%가 1분 이내에 적어도 한 개 채널로 전달된다.
- **SC-03**: 평가 리마인더 도입 후 마감 내 동료 평가 제출률이 기존 대비 30% 이상 향상된다.
- **SC-04**: 사용자 알림 설정 변경이 100% 즉시 다음 알림부터 반영된다.
- **SC-05**: 중복 알림(동일 이벤트 5분 내 재전송) 비율이 전체 알림의 1% 미만이다.
- **SC-06**: 알림으로 인한 사용자 이탈/수신 거부(전체 off) 비율이 월 5% 미만으로 유지된다.

## 5. Assumptions

- 사용자는 가입 시 유효한 이메일을 등록하며, 푸시는 사용자가 명시적으로 허용한 경우에만 사용한다.
- 알림 종류는 001 명세의 이벤트(매칭·협업·평가·운영)를 기반으로 하며, 추후 종류 추가가 가능하도록 확장형으로 설계한다.
- 야간 묶음 발송 기본 시간대는 22:00~08:00, 알림 보관 기간 기본값은 90일로 가정한다(설정 가능).
- 필수 알림(보안·계정·법적 고지)의 구체 목록은 운영 정책으로 별도 관리한다.

## 6. Dependencies

### 6.1 외부 시스템 의존성
- 이메일 발송 서비스(트랜잭션 메일)
- 푸시 발송 서비스(웹/모바일 푸시)
- 001 플랫폼의 이벤트 소스(매칭·협업 분석·평가·운영 모듈)

### 6.2 조직·정책 의존성
- 필수 알림 종류 및 야간/보관 기본값에 대한 운영 정책 승인

---

## 7. Out of Scope (1차 출시 제외 범위)

- SMS·카카오톡 등 추가 채널
- 사용자 정의 알림 규칙(커스텀 트리거)
- 알림 A/B 테스트 및 발송 최적화 자동화
- 다국어 알림 템플릿(1차는 한국어)

## 8. Open Questions (Clarification 필요)

> `/speckit.clarify` 단계에서 확정 예정. 아래 3개는 본 명세 작성 시 정한 기본값이 있으나, 범위·사용자 경험에 영향이 커 확인이 필요하다.

- **Q1 (채널 범위)**: 1차 출시에서 푸시 알림을 포함하는가, 아니면 인앱+이메일만으로 시작하는가?
- **Q2 (실시간성)**: 인앱 알림이 실시간 푸시(접속 중 자동 표시)되어야 하는가, 아니면 화면 새로고침/주기 조회로 충분한가?
- **Q3 (수신 대상 단위)**: 협업 위험 신호 알림을 팀장에게만 보내는가, 팀 전체에 보내는가?
