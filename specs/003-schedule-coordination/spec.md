## 1. 개요 (Overview)

UniTeam에서 팀이 확정된 후, 팀원들의 **가용 시간을 자동 비교해 공통 가능한 회의 시간을 제안**하고, 회의·마감·작업 일정을 한곳에서 관리하며, 개인 외부 캘린더(Google Calendar)와 동기화하는 기능이다. 일정 변경·임박 시 002 알림 시스템을 통해 리마인더를 전송한다. 기존 회원 프로필의 주간 가용시간(`availabilities`) 데이터를 기반으로 한다.

본 명세서는 비즈니스 이해관계자(학생, 팀장)가 검토할 수 있도록 **사용자 가치와 시스템이 해야 할 일**에 초점을 맞추며, 구현 기술 스택은 다루지 않는다.

## 2. User Scenarios & Testing *(mandatory)*

### 2.1 Primary User Story

**김민서(팀장) 시점**

팀이 확정됐지만 5명의 시간표가 제각각이라 첫 회의 잡기가 늘 골치였다.

1. 민서가 프로젝트 일정 탭에 들어가면, 시스템이 팀원 5명의 주간 가용시간을 겹쳐 **공통 가능 시간 후보를 겹침 인원·길이 순으로 제안**한다.
2. 민서는 "화요일 20:00–21:00 (5명 전원 가능)" 후보를 선택해 첫 회의를 생성한다.
3. 팀원들에게 회의 일정 알림이 가고, 각자 **참석/불참(RSVP)**을 응답한다.
4. 민서가 프로젝트 마감일과 중간 산출물 일정을 등록하면 팀 캘린더에 표시된다.
5. 각 팀원은 자신의 **Google Calendar와 동기화**를 켜서, 팀 회의·마감이 개인 캘린더에 자동 반영되게 한다.
6. 회의 1시간 전, 마감 하루 전에 리마인더 알림이 전송된다.
7. 한 팀원이 시간이 안 맞아 회의 시간을 변경 제안하자, 시스템이 대체 공통 시간 후보를 다시 제시하고 변경 시 전원에게 알린다.

### 2.2 Acceptance Scenarios

| # | Given (전제) | When (행위) | Then (기대 결과) |
|---|---|---|---|
| AS-01 | 팀이 확정(2명 이상 ACCEPTED)됨 | 일정 탭에 진입함 | 팀원 가용시간을 겹친 공통 시간 후보가 겹침 인원·길이 순으로 표시된다 |
| AS-02 | 공통 시간 후보가 표시됨 | 팀장이 후보 하나로 회의를 생성함 | 회의가 저장되고 팀원에게 회의 일정 알림이 전송된다 |
| AS-03 | 회의가 생성됨 | 팀원이 참석/불참으로 응답함 | RSVP가 기록되고 참석 현황이 집계된다 |
| AS-04 | 팀장이 마감·산출물 일정을 등록함 | 일정 항목을 저장함 | 팀 캘린더에 표시되고 관련 팀원에게 공유된다 |
| AS-05 | 사용자가 Google Calendar 연동을 켬 | 외부 캘린더 연결을 승인함 | 팀 회의·마감이 사용자 개인 캘린더에 생성·동기화된다 |
| AS-06 | 회의·마감 시점이 임박함 | 리마인더 기준 시점(회의 1시간 전·마감 1일 전)이 도래함 | 해당 팀원에게 리마인더 알림이 전송된다 |
| AS-07 | 팀원이 회의 시간 변경을 제안함 | 변경 요청을 제출함 | 시스템이 대체 공통 시간 후보를 제시하고, 확정 시 전원에게 변경 알림이 전송된다 |
| AS-08 | 공통 가능 시간이 없음 | 일정 탭에서 후보를 조회함 | 시스템이 "전원 공통 시간 없음"을 알리고 최다 인원이 가능한 차선 후보를 제시한다 |

### 2.3 Edge Cases

- **EC-01 공통 시간 없음**: 전원이 겹치는 슬롯이 없으면 최다 인원 가능 슬롯을 "N/총원" 표기로 제시한다.
- **EC-02 시간대(타임존) 상이**: 팀원 타임존이 다르면 회의 시간은 각 사용자 로컬 시간으로 표시한다(저장은 UTC 기준).
- **EC-03 가용시간 미입력 팀원**: 가용시간을 입력하지 않은 팀원은 후보 계산에서 "미정"으로 분류되고 가입 시 입력을 유도한다.
- **EC-04 외부 캘린더 연동 해제·토큰 만료**: 동기화 토큰이 만료되면 동기화를 보류하고 사용자에게 재연동을 안내한다(이미 생성된 일정은 유지).
- **EC-05 회의 시간 충돌**: 동일 사용자의 두 회의가 시간상 겹치면 충돌 경고를 표시한다.
- **EC-06 멤버 변동**: 회의 생성 후 팀원이 추가/이탈하면 참석 대상과 공통 시간 후보가 갱신된다.
- **EC-07 과거 시간 일정**: 과거 시각으로의 회의/마감 생성은 거부한다.
- **EC-08 반복 회의**: 매주 같은 시간 반복 회의는 단일 생성으로 N주간 인스턴스를 만들고, 개별 회차 취소가 가능하다.

## 3. Requirements *(mandatory)*

### 3.1 Functional Requirements

#### FR-A. 공통 시간 계산
- **FR-A1**: 시스템은 프로젝트의 ACCEPTED 팀원 주간 가용시간을 교집합 계산해 공통 가능 슬롯을 산출해야 한다.
- **FR-A2**: 시스템은 공통 슬롯을 겹침 인원 수와 슬롯 길이 기준으로 정렬해 제시해야 한다.
- **FR-A3**: 전원 공통 슬롯이 없으면 최다 인원 가능 슬롯을 차선으로 제시해야 한다(EC-01).
- **FR-A4**: 최소 회의 길이(기본 30분) 미만 슬롯은 후보에서 제외해야 한다.

#### FR-B. 회의·일정 관리
- **FR-B1**: 팀장은 공통 시간 후보 또는 직접 지정으로 회의를 생성할 수 있어야 한다.
- **FR-B2**: 시스템은 회의·마감·산출물 일정 유형을 지원하고 팀 캘린더에 표시해야 한다.
- **FR-B3**: 팀원은 회의에 참석/불참(RSVP)을 응답할 수 있고 시스템은 참석 현황을 집계해야 한다.
- **FR-B4**: 시스템은 반복 회의(주간) 생성과 개별 회차 취소를 지원해야 한다(EC-08).
- **FR-B5**: 시스템은 과거 시각 일정 생성을 거부해야 한다(EC-07).

#### FR-C. 외부 캘린더 동기화
- **FR-C1**: 사용자는 Google Calendar 연동을 켜고 끌 수 있어야 한다.
- **FR-C2**: 연동 시 시스템은 팀 회의·마감을 사용자 개인 캘린더에 생성·갱신·삭제 동기화해야 한다.
- **FR-C3**: 동기화 토큰 만료·연동 해제 시 동기화를 중단하고 재연동을 안내해야 한다(EC-04).

#### FR-D. 리마인더·알림 (002 연동)
- **FR-D1**: 시스템은 회의 임박(기본 1시간 전)·마감 임박(기본 1일 전) 시 대상 팀원에게 리마인더를 전송해야 한다.
- **FR-D2**: 일정 생성·변경·취소 시 관련 팀원에게 알림을 전송해야 한다(002 `SCHEDULE_CHANGE`).
- **FR-D3**: 리마인더 시점은 사용자/팀 설정으로 조정 가능해야 한다.

#### FR-E. 타임존
- **FR-E1**: 일정은 UTC로 저장하고 각 사용자 로컬 타임존으로 표시해야 한다(EC-02).

### 3.3 Key Entities

| 엔티티 | 설명 | 주요 속성 |
|---|---|---|
| **ScheduleEvent** | 회의·마감·산출물 일정 | id, projectId, type(MEETING/DEADLINE/MILESTONE), title, startsAt(UTC), endsAt, createdBy, recurrenceRule |
| **EventRsvp** | 회의 참석 응답 | eventId, userId, response(ATTEND/DECLINE/PENDING) |
| **CommonSlot** (계산 결과, 비영속) | 공통 가능 슬롯 | weekday, startMin, endMin, availableCount, totalMembers |
| **CalendarConnection** | 외부 캘린더 연동 | userId, provider(GOOGLE), accessToken(암호화), refreshToken, syncState, expiresAt |
| **CalendarSyncMap** | 내부 일정 ↔ 외부 이벤트 매핑 | scheduleEventId, userId, externalEventId, lastSyncedAt |

## 4. Success Criteria *(mandatory)*

### 4.1 Measurable Outcomes

- **SC-01**: 팀이 공통 회의 시간 후보를 받아 첫 회의를 잡기까지 평균 3분 이내.
- **SC-02**: 5인 팀 기준 공통 시간 후보 계산 결과가 2초 이내에 표시된다.
- **SC-03**: 일정 조율 기능 도입 후 "회의 시간 조율 어려움" 관련 불만이 기존 대비 40% 이상 감소.
- **SC-04**: Google Calendar 연동 사용자의 팀 회의 90% 이상이 개인 캘린더에 정확히 반영된다.
- **SC-05**: 회의 리마인더 수신자의 회의 참석률이 미수신 대비 20% 이상 높다.

## 5. Assumptions

- 회원은 프로필에서 주간 가용시간(`availabilities`)을 입력하며, 본 기능은 이를 1차 입력원으로 사용한다.
- 1차 외부 캘린더는 Google Calendar만 지원한다(타 제공자는 범위 외).
- 리마인더 기본값: 회의 1시간 전, 마감 1일 전(설정 가능).
- 일정 저장은 UTC, 표시는 사용자 타임존(기본 Asia/Seoul).
- 회의 생성·일정 등록 권한은 팀장(프로젝트 owner) 기본, 팀원 권한 부여는 추후.

## 6. Dependencies

### 6.1 외부 시스템 의존성
- Google Calendar API (OAuth 2.0, 캘린더 읽기/쓰기 스코프)
- 002 알림 시스템(리마인더·변경 알림 전달)
- 001 `availabilities`·`project_members` 데이터

### 6.2 정책 의존성
- 외부 캘린더 토큰 저장·암호화 및 권한 범위에 대한 개인정보 정책

---

## 7. Out of Scope (1차 출시 제외 범위)

- Google 외 캘린더(Outlook/Apple/Naver) 연동
- 양방향 동기화(외부 캘린더의 개인 일정을 UniTeam 가용시간에 자동 반영)
- 회의실·물리 자원 예약
- 화상회의 링크 자동 생성(Zoom 등 — 별도 협업도구 연동 기능)

## 8. Open Questions (Clarification 필요)

> `/speckit.clarify`에서 확정 예정. 기본값은 정해두었으나 범위·UX 영향이 커 확인 필요.

- **Q1 (가용시간 입력원)**: 공통 시간 계산을 주간 반복 가용시간(`availabilities`)만으로 하는가, 아니면 특정 주의 단발성 가용 입력도 받는가?
- **Q2 (캘린더 동기화 방향)**: 1차는 UniTeam → Google 단방향만 하는가, 외부 일정을 읽어 충돌 표시까지 하는가?
- **Q3 (회의 생성 권한)**: 회의/일정 생성을 팀장만 허용하는가, ACCEPTED 팀원 누구나 가능한가?
